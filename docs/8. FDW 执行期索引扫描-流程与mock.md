# FDW 执行期索引扫描：流程与 mock 验证

本文描述从 `BeginForeignScan` 起的 FDW 执行器**向量索引扫描**流程、规划期 fdw_private 的 mock 接入方式，及在 openGauss 内经真实 MinIO + REST catalog 的端到端验证步骤。执行器实现见 `DataInfraLab/iceberg_fdw`（`edcffea`），规划期成本/投影改造见 `bc46bee`、`2118eae`。

本文供其他环境的协作者复现，第 5 节为完整步骤。配套文件（同目录）：

| 文件 | 用途 |
|---|---|
| `8-fdw-mock-index-shim.patch` | FDW 规划期 mock shim，`git apply` 到 iceberg_fdw 仓（基线 main `2118eae`）后重建 FDW |
| `8-idx-e2e-wiring.sql` | 库内接线 + 建索引 + 扫描脚本，用 `-v` 传环境值 |

**前置条件**：iceberg_fdw / new-catalog / iceberg-rust-bridge 三仓已按各自方式构建，openGauss 已安装 `iceberg_catalog` + `iceberg_fdw` 扩展，本机可运行 docker（起 MinIO + REST）。

**环境适配点**（按本机替换）：gaussdb 端口与连接方式、三仓路径、Rust 工具链、MinIO/REST 端点与凭证（用下述 compose 时为默认值）。文中命令以 compose 默认端点 `127.0.0.1:9000`（MinIO）/`127.0.0.1:8181`（REST）、凭证 `admin`/`password` 为例。

## 1. 范围与模型

FDW 扫描模型是正交 2×2：**{标量, 向量}** 是列属性，**{全表, 索引}** 是扫描方法。两个"全表"格合一，落地为 3 条执行器→bridge 路径：

| 路径 | bridge 接口 | 投影列 | 现状 |
|---|---|---|---|
| 全表扫描 | `iceberg_bridge_scan_open` | 传列名投影，bridge 按名+序返回 | 已通 |
| 向量索引扫描 | `iceberg_index_rs_search_vector_by_metadata` + `metadata_scan_*` | 不传列名，返回「表全部列 + 尾部 `_distance`」 | 经真实后端 e2e 已通（本文） |
| 标量索引扫描 | 无 bridge 数据面 | — | 执行器占位 `ereport`，规划期发现恒 false |

本文只覆盖向量索引扫描——它是当前唯一具备 bridge 数据面的索引扫描。其 bridge 数据面（`build→search→scan`）由 bridge 仓自带 `tests/cpp/index_abi_e2e.cpp` 经真实 MinIO + REST 验证；本文在其上把同一条数据面穿过 FDW 执行器跑通。

## 2. 执行期流程（BeginForeignScan → 行）

```
BeginForeignScan
  ├─ icebergDeserializeScanPrivate(fdw_private)：scan_method=VECTOR_INDEX 时取 index 子表
  │    metadata_location / namespace / table_name / snapshot_id 均来自 fdw_private
  │    （规划期由 GetForeignRelSize 经 catalog 取得后下传，执行期不再查 catalog）
  ├─ 组 IcebergIndexScanRequest
  └─ iceberg_index_scan_open
        ├─ dlopen libiceberg_rust_bridge.so（ICEBERG_RUST_BRIDGE_SO / _HOME / ld 路径）
        ├─ dlsym 索引符号 + 校验 abi_version == 3
        ├─ iceberg_bridge_storage_open(storage_config_json)
        └─ iceberg_index_rs_search_vector_by_metadata(storage, request) → IcebergIndexScan*

IterateForeignScan（循环）
  ├─ iceberg_index_scan_next → iceberg_index_rs_metadata_scan_next_batch → ArrowArray
  └─ iceberg_arrow_materialize_projection_row：按 projected_columns 列序映射子数组 → TupleTableSlot，丢弃尾部 _distance

ReScanForeignScan → iceberg_index_rs_metadata_scan_rescan（重放）
EndForeignScan    → metadata_scan_close → storage_release → 删 MemoryContext
```

关键约定：

- **定位**：索引由 `(metadata_location, index_name)` 定位，`metadata_location` 已 pin 当前 snapshot，执行器不再单独传 snapshot。
- **元数据来源**：执行期 `metadata_location` 等取自 `fdw_private`，由规划期 `GetForeignRelSize` 调 `iceberg_catalog_get_table_info(relid)` 读 `iceberg_catalog.tables_internal` 后下传。
- **不投影**：v3 search 请求无 projection 字段，输出固定为「表全部列 + 尾部 `_distance`」，列序 = 表 schema 序。
- **列序对齐**：执行器把 `projected_columns` 喂成整表全列、按 attnum/schema 列序，与 bridge 输出逐列对齐，复用与全表扫描相同的 `iceberg_arrow_materialize_projection_row`，丢弃尾部 `_distance`。此对齐要求外表列序与 Iceberg schema 列序一致（建表时保证）。
- **失败可见**：storage_open / search / next_batch / rescan 任一返回非 OK，立即 `ereport(ERROR)` 携带 bridge 错误消息，不静默回退全表扫描。

`IcebergIndexScanRequest` 字段（执行器→适配器内部契约）：

| 字段 | 来源 | 说明 |
|---|---|---|
| storage_config_json | server 选项 | 与全表扫描同一份 |
| metadata_location | fdw_private（规划期取自 catalog） | v3 定位键 |
| namespace_name / table_name | fdw_private | 组 `["ns"]` JSON 数组 + 表名 |
| index_name | fdw_private | 规划期下传 |
| vector_column_name | fdw_private | 信息性，v3 search 不传列名 |
| query_vector / query_vector_len | fdw_private | 查询向量 |
| metric / k / fetch_k | fdw_private | 距离类型、近邻数；fetch_k 暂取 k |
| projected_columns | 表描述符 | 整表全列、列序物化用 |

## 3. 规划期与 fdw_private mock

### 3.1 规划期现状

`GetForeignRelSize` 把带 magic 的 `IcebergFdwPlanState`（catalog 元数据 + 成本估算 + 列映射）写入 `baserel->fdw_private`；`GetForeignPaths` 只产一条全表扫描 path 并把 `plan_state->path_info` 置 NULL；`GetForeignPlan` 据 `path_info` 分流：非空且为 VECTOR_INDEX 走 `icebergBuildIndexScanPrivate`，否则全表。当前真实索引发现 seam 为空（`path_info` 恒 NULL），向量索引路径靠 mock 触发。

### 3.2 option 门控的 fdw_private mock

`icebergTryBuildMockIndexPath`（`iceberg_fdw.cpp`）在 `GetForeignPlan` 内读外表 OPTIONS，命中 `mock_index_scan='vector'` 时合成 `IcebergFdwIndexPathInfo` 并优先于 `plan_state->path_info`，使普通 `SELECT` 进入向量索引执行器。门控关闭（无该 option）时全表路径零影响。

| option | 含义 |
|---|---|
| `mock_index_scan` | 取 `vector` 时启用 |
| `mock_index_name` | 索引名，与 `create_index` 一致 |
| `mock_vector_column` | 向量列名 |
| `mock_query_vector` | 逗号分隔的查询向量，如 `1,0` |
| `mock_topk` | 近邻数 k（fetch_k 取同值） |

命中后规划期序列化进 `fdw_private`，布局：

```
fdw_private = [ Integer(1=VECTOR_INDEX), [index_name, vector_column_name, metric, k, fetch_k, query_vector(List<Float>)] ]
```

执行器 `icebergDeserializeIndexScanPrivate` 按此布局反序列化。该布局是规划期与执行器之间的稳定契约。

### 3.3 真实上游待接

真实链路由内核 VectorSearch 算子从 `ORDER BY vec <-> const LIMIT k` 取 query_vector/k，连同索引选择注入规划期。注意 `baserel->fdw_private` 已被 `IcebergFdwPlanState` 占用，内核须经一条不覆盖该字段的 seam 注入（或与 plan_state 共存）。mock shim 标 TODO，待此 seam 就绪后移除。

## 4. 下游：真实 MinIO + REST catalog

向量索引引擎只支持 `memory://` 与 REST catalog，不支持 `file://`，故索引数据面用 MinIO + REST。

- **基础设施**：`docker compose -f iceberg-rust-bridge/tests/e2e_infra/docker-compose.yml up -d` 起 MinIO（`:9000/:9001`）+ REST fixture（`:8181`），凭证 admin/password，桶 `warehouse`。健康检查 `curl :9000/minio/health/live`、`curl :8181/v1/config`。
- **供数**：`cargo run --example e2e_provision`（bridge 仓，需 `env -u LD_LIBRARY_PATH PATH=~/.cargo/bin:$PATH`）在 REST catalog 建 `e2e_cpp.vectors`（`id int`、`embedding list<float>` dim2，4 行 `[0,0]/[1,0]/[3,0]/[1,1]`），打印 `metadata_location`。
- **建索引**：`iceberg_catalog.create_index` 一步完成 `iceberg_index_rs_build_index_by_metadata`（与 `index_abi_e2e` 同一 C-ABI）→ `iceberg_bridge_update_statistics` → CAS 改 `tables_internal.metadata_location` → INSERT `table_indexes`。
- **实例环境**：`create_index` 经 `open_iceberg_storage()` 读环境变量定位存储，故 gaussdb 须以 `ICEBERG_WAREHOUSE=s3://warehouse/` + `ICEBERG_S3_ENDPOINT/ACCESS_KEY/SECRET_KEY/REGION` 启动；FDW 扫描则用外表 server 的 s3 选项。

## 5. 复现步骤（异构环境）

**① 打 mock shim 并重建 FDW**：在 iceberg_fdw 仓根 `git apply <本目录>/8-fdw-mock-index-shim.patch`，按本仓常规方式重建并安装 `iceberg_fdw.so`。shim 改 `iceberg_fdw.cpp`：注册 5 个 `mock_*` 外表 option，并在 `GetForeignPlan` 加 `icebergTryBuildMockIndexPath`。

**② 起基础设施**：`docker compose -f iceberg-rust-bridge/tests/e2e_infra/docker-compose.yml up -d`，待 `curl 127.0.0.1:9000/minio/health/live` 与 `curl 127.0.0.1:8181/v1/config` 通。

**③ 供数**：iceberg-rust-bridge 仓 `cargo run --example e2e_provision`，记录输出的 `metadata_location`（记为 `<base_metadata>`）。若 shell 注入了 openGauss 旧 `libstdc++`，前置 `env -u LD_LIBRARY_PATH PATH=~/.cargo/bin:$PATH`。

**④ 以 s3 环境重启 gaussdb**：导出后重启，使 `catalog.create_index` 的 `open_iceberg_storage()` 定位到 MinIO：

```
ICEBERG_WAREHOUSE=s3://warehouse/
ICEBERG_S3_ENDPOINT=http://127.0.0.1:9000
ICEBERG_S3_ACCESS_KEY=admin
ICEBERG_S3_SECRET_KEY=password
ICEBERG_S3_REGION=us-east-1
```

**⑤ 接线 + 建索引 + 扫描**：在全新 database 内执行 `8-idx-e2e-wiring.sql`：

```
gsql -d <db> -p <port> \
  -v base_metadata="'<base_metadata>'" \
  -v s3_endpoint="'http://127.0.0.1:9000'" \
  -v s3_key="'admin'" -v s3_secret="'password'" \
  -f 8-idx-e2e-wiring.sql
```

脚本依次：建 namespace + 表（schema 须与 e2e_provision 一致）→ 给自动建的 `iceberg_catalog_server` 补 s3 凭证 → 重建外表为 `vector(2)` + `mock_*` 选项 → `UPDATE ... relid` → `UPDATE ... metadata_location=<base_metadata>` → `iceberg_catalog.create_index`（build→update_statistics→CAS 改 metadata_location→INSERT table_indexes）→ `EXPLAIN` + 实扫。

外表 namespace 与供数器同名（`e2e_cpp`）时，`create_table` 会向同一 s3 目录另写一份占位元数据，随后改指数据元数据，互不覆盖（文件按 uuid 命名）。

## 6. 验证结果

`EXPLAIN`：

```
 Foreign Scan on e2e_cpp.vectors
   Iceberg FDW: bridge vector index scan
   Projection Columns: id,embedding
   Index Name: idx_embedding
   Metric: L2
   TopK: 4 / FetchK: 4 / Query Vector Dim: 2
```

实扫（query `[1,0]`，L2，距离含 sqrt）按近邻序返回：

| id | embedding | L2 距离 |
|---|---|---|
| 2 | [1,0] | 0 |
| 1 | [0,0] | 1 |
| 4 | [1,1] | 1 |
| 3 | [3,0] | 2 |

`create_index` 返回 `{"success": true}`，`table_indexes` 注册 `idx_embedding`（`field_ids={2}`, status active）。链路 `SELECT → BeginForeignScan → iceberg_index_scan_open → search_vector_by_metadata → materialize` 端到端成立，`_distance` 列被丢弃。
