# FDW 执行期索引扫描：流程与 mock 验证

本文描述从 `BeginForeignScan` 起的 FDW 执行器**向量索引扫描**流程，及其上下游 mock 方式、文件与运行步骤。对应执行器实现已合入 `DataInfraLab/iceberg_fdw` main（commit `edcffea`）。

## 1. 范围与模型

FDW 扫描模型是正交 2×2：**{标量, 向量}** 是列属性，**{全表, 索引}** 是扫描方法。两个"全表"格合一，落地为 3 条执行器→bridge 路径：

| 路径 | bridge 接口 | 投影列 | 现状 |
|---|---|---|---|
| 全表扫描 | `iceberg_bridge_scan_open` | 传列名投影，bridge 按名+序返回 | 已通 |
| 向量索引扫描 | `iceberg_index_rs_search_vector_by_metadata` + `metadata_scan_*` | 不传列名，返回「表全部列 + 尾部 `_distance`」 | 本文对象 |
| 标量索引扫描 | 无 bridge 数据面 | — | 执行器占位，规划期发现恒 false |

本文只覆盖向量索引扫描——它是当前唯一具备 bridge 数据面的索引扫描。

## 2. 执行期流程（BeginForeignScan → 行）

流程图见 [fdw_index_scan_exec_flow.svg](fdw_index_scan_exec_flow.svg)。文字描述：

```
BeginForeignScan
  ├─ 反序列化 fdw_private（scan_method=VECTOR_INDEX 时取 index 子表）
  ├─ iceberg_catalog_get_table_info(relid) → metadata_location（读本地表 iceberg_catalog.tables_internal）
  ├─ 组 IcebergIndexScanRequest
  └─ iceberg_index_scan_open
        ├─ dlopen libiceberg_rust_bridge.so（ICEBERG_RUST_BRIDGE_SO / _HOME / ld 路径）
        ├─ dlsym 9 符号 + 校验 abi_version == 3
        ├─ iceberg_bridge_storage_open(storage_config_json)
        └─ iceberg_index_rs_search_vector_by_metadata(storage, request) → IcebergIndexScan*

IterateForeignScan（循环）
  ├─ iceberg_index_scan_next → iceberg_index_rs_metadata_scan_next_batch → ArrowArray
  └─ iceberg_arrow_materialize_projection_row：按 projected_columns 列序映射子数组 → TupleTableSlot，丢弃尾部 _distance

ReScanForeignScan → iceberg_index_rs_metadata_scan_rescan（重放）
EndForeignScan    → metadata_scan_close → storage_release → 删 MemoryContext
```

关键约定：

- **定位**：索引由 `(metadata_location, index_name)` 定位，`metadata_location` 已 pin 当前 snapshot，执行器不再单独传 snapshot。
- **不投影**：v3 search 请求无 projection 字段，输出固定为「表全部列 + 尾部 `_distance`」，列序 = 表 schema 序。
- **列序对齐**：执行器把 `projected_columns` 喂成整表全列、按 attnum/schema 列序（`icebergBuildProjectedColumns`），与 bridge 输出逐列对齐，复用与全表扫描相同的 `iceberg_arrow_materialize_projection_row`，丢弃尾部 `_distance`。此对齐前提要求外表列序与 Iceberg schema 列序一致（由 Catalog 建表时保证）。
- **失败可见**：storage_open / search / next_batch / rescan 任一返回非 OK，立即 `ereport(ERROR)` 携带 bridge 错误消息，不静默回退全表扫描。

`IcebergIndexScanRequest` 字段（执行器→适配器内部契约）：

| 字段 | 来源 | 说明 |
|---|---|---|
| storage_config_json | server 选项 | 与全表扫描同一份 |
| metadata_location | catalog tables_internal | v3 定位键 |
| namespace_name / table_name | catalog | 组 `["ns"]` JSON 数组 + 表名 |
| index_name | fdw_private | 规划期下传 |
| vector_column_name | fdw_private | 信息性，v3 search 不传列名 |
| query_vector / query_vector_len | fdw_private | 查询向量 |
| metric / k / fetch_k | fdw_private | 距离类型、近邻数；fetch_k 暂无 ABI over-fetch，取 k |
| projected_columns | 表描述符 | 整表全列、列序物化用 |

## 3. 上游 mock（fdw_private 及以上）

规划期产出向量索引计划的触发点是 `baserel->fdw_private` 被置为 `IcebergFdwIndexPathInfo`：

- **真实链路**：kernel VectorSearch 把查询向量 q 与索引选择写入 `baserel->fdw_private`（`GetForeignRelSize`/`GetForeignPaths` 前）。这是 Andy 对接的接点。
- **mock 方式**：用一个 debug GUC（例如 `iceberg_fdw.mock_vector_index_name` + `iceberg_fdw.mock_query_vector`）在 `GetForeignRelSize` 内构造 `IcebergFdwIndexPathInfo{valid=true, index_name, vector_column_name, query_vector, k, metric}`，使普通 `SELECT` 进入向量索引路径。该 GUC 仅为执行期联调，作为 overlay 存放于本仓，不进入 FDW 主干。

命中后规划期把它序列化进 `fdw_private`（`icebergBuildIndexScanPrivate`），布局：

```
fdw_private = [ Integer(1=VECTOR_INDEX), [index_name, vector_column_name, metric, k, fetch_k, query_vector(List<Float>)] ]
```

执行器 `icebergDeserializeIndexScanPrivate` 按此布局反序列化。该布局是规划期与执行器之间的稳定契约，本次执行器改造未改动它。

## 4. 下游 mock（bridge v3）

无 docker/MinIO 时无法造真实索引，用 mock bridge `.so` 替身驱动执行器数据面。文件在 [../iceberg_fdw/index_exec_mock/](../iceberg_fdw/index_exec_mock/)：

| 文件 | 作用 |
|---|---|
| `mock_index_bridge.c` | 编成 `libmock_index_bridge.so`，导出适配器 dlsym 的全部 9 个符号；`search_vector_by_metadata` 返回固定 Arrow batch：`[id int32, label int32, _distance float64]` × 4 行，按距离升序 |
| `index_exec_harness.c` | 复现 `index_scan_adapter` 的 v3 调用序列（用同一套 `iceberg_index_abi.h`/`iceberg_bridge_abi.h`）：dlopen→dlsym 9→abi_version→storage_open→search→next_batch 循环解码→rescan 重放→close |
| `run.sh` | 一键构建并跑 A（mock）/ B（真实 bridge）两场景 |

mock 返回的列布局「表全部列 + 尾部 `_distance`」与真实 v3 同形：对应外表 `(id int4, label int4)`，materialize 取前 2 列、丢弃 `_distance`。

运行（编译需用 `env -u LD_LIBRARY_PATH` 避开 openGauss 旧 libstdc++）：

```bash
cd iceberg_fdw/index_exec_mock && ./run.sh
```

## 5. 验证结果

**A) 对 mock bridge——执行器数据面跑通**：

```
[ok] dlsym 9/9 符号命中
abi_version = 3 (expect 3) -> OK
storage_open -> OK
search_vector_by_metadata -> OK
Iterate（首次）: 4 行 [id=10..13, label=100..103, _distance=0.5..3.5]
ReScan 后重放: 4 行（与首次一致）
```

**B) 对真实 bridge——ABI 接缝 + 错误契约**：

```
[ok] dlsym 9/9 符号命中
abi_version = 3 (expect 3) -> OK
storage_open -> OK
search_vector_by_metadata -> ERR
  load table: ... Failed to read file /tmp/does-not-exist/v1.metadata.json: No such file (os error 2)
```

harness 与执行器适配器共用同一套头文件。B 中真实 bridge 解析了整个请求结构体、走到加载表才因文件不存在干净报错（而非 INVALID_ARGUMENT 或崩溃），证明结构体布局、符号名、ABI 版本、错误契约与真实 v3 一致——即执行器对 v3 的对接正确。A 证明 search→next_batch→Arrow 解码→行→rescan 的数据面闭环成立。

## 6. 在 openGauss 内 SQL 级实跑（配方）

本地无 docker/MinIO，真实向量索引无法构建，故下列 SQL 级实跑步骤为复现配方（A/B 已用 harness 实跑验证执行器调用序列与数据面）。两端 mock 就绪后可在库内直跑：

1. 让 gaussdb 的 `ICEBERG_RUST_BRIDGE_SO` 指向 `libmock_index_bridge.so` 后重启（mock 仅实现索引扫描所需符号）。
2. 建 foreign server（含 `warehouse` 及占位 s3 选项）与 foreign table `(id int4, label int4)`。
3. 向 `iceberg_catalog.tables_internal` 直插一行：`relid`=外表 oid、`namespace`/`table_name`、`metadata_location`=任意占位（mock 忽略）。
4. 启用上游 mock GUC，使该外表的 `SELECT` 进入向量索引路径。
5. `EXPLAIN SELECT * FROM ft;` 应显示 `bridge vector index scan`；`SELECT * FROM ft;` 返回 mock 的 4 行（_distance 已被丢弃）。

真实数据路径（替代 mock bridge）：起 MinIO + REST catalog（`iceberg-rust-bridge/tests/e2e_infra/docker-compose.yml`），经 bridge `build_index_by_metadata` → `update_statistics` → 发布得到带索引的 `metadata_location`，再走步骤 2–5。
