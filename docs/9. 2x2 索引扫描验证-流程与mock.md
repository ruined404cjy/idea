# 2×2 索引扫描验证：流程与 mock

## 1. 目标与范围

2×2 指 `{标量, 向量} × {全表, 索引}` 四条 FDW 扫描路径。本文给出在**单一 fs 环境**（无需 MinIO/REST）连续验证四条路径的可复现流程。

验证对象是 FDW 执行期：`BeginForeignScan → bridge`。规划期的索引发现尚未落地，由 option 门控的 mock shim 旁路合成 `fdw_private` 驱动索引路径。执行期只从 `fdw_private` 取数，与产出方（mock 或将来的规划期）无关。

## 2. 路径与改动边界

| 路径 | 执行期请求结构 | 谓词/查询载体 | 改动归属 |
|---|---|---|---|
| ① 标量 × 全表 | `IcebergSdkScanRequest` | `filter_json`（iceberg-Predicate `{"Binary":{"op":"Eq",…}}`） | 基线，未改 |
| ② 标量 × 索引 | `IcebergScalarIndexScanRequest` | `expression_json`（bridge 标量 `{"field_id","op","value"}`） | A：已并 FDW main |
| ③ 向量 × 全表 | `IcebergSdkScanRequest`（内核 VectorSearch 在其上做 TopK） | 无下推 | 基线，未改 |
| ④ 向量 × 索引 | `IcebergIndexScanRequest` | `query_vector` + k | A：已并 FDW main |

①③ 的谓词格式（iceberg-Predicate）与 ②的标量表达式分属不同引擎，互不混用。索引路径（②④）在 FDW main 上**默认休眠**，须由 mock shim（B）合成 path 才触发。

## 3. 依赖

| 组件 | 要求 |
|---|---|
| iceberg_fdw | `main` 已含标量/向量索引执行器（A）。本验证另需打 mock 补丁（B）。 |
| iceberg-rust-bridge | 提供 `provision_local`、`file_scan_vector` 供数与运行时 `.so`（导出 `iceberg_index_rs_search_{vector,scalar}_by_metadata` + 共享 `metadata_scan_*`，`abi_version==3`）。 |
| openGauss 内核 | 含 catalog/FDW 扩展。**注意**：FDW main 的向量发现依赖内核 PR#3 的 `root->vectorsearch_topk_infos`。 |

### 3.1 内核 PR#3 与构建

FDW main 含向量发现代码，引用内核 `root->vectorsearch_topk_infos` / `VectorSearchTopKInfo`。若本机内核**尚无**该字段，FDW main 直接编译失败（`'VectorSearchTopKInfo' does not name a type`）。两种处理：

- 内核已合入 PR#3：直接构建。
- 内核无 PR#3：先应用 `9-fdw-andy-pr3-stub.patch`，将向量发现函数置空（返回 NULL），向量发现转由 mock shim 承担，仅用于本验证；内核合入后勿用。

## 5. 复现步骤

### ① 构建并安装 FDW

```
cd <iceberg_fdw>
git checkout main && git pull
git apply <docs>/9-fdw-2x2-mock-shim.patch
# 内核无 PR#3 时追加：
git apply <docs>/9-fdw-andy-pr3-stub.patch

PGC=$GAUSSHOME/bin/pg_config ; SRC=<kernel>/src/include
env -u LD_LIBRARY_PATH make PG_CONFIG=$PGC OPENGAUSS_SRC_INCLUDE=$SRC -j4
env -u LD_LIBRARY_PATH make PG_CONFIG=$PGC OPENGAUSS_SRC_INCLUDE=$SRC install
```

### ② 以 fs 环境重启 gaussdb

```
export GAUSSHOME=<...> PGDATA=<...>
export PATH=$GAUSSHOME/bin:$PATH LD_LIBRARY_PATH=$GAUSSHOME/lib
export ICEBERG_WAREHOUSE=file:///home/omm/iw_pg
export ICEBERG_RUST_BRIDGE_SO=<bridge>/target/release/libiceberg_rust_bridge.so
gs_ctl restart -D $PGDATA -Z single_node
```

FDW 扫描的存储配置取自外表 server 选项（warehouse=`file://` → fs），与 env 无关；env 仅供 catalog DDL。

### ③ 供数：标量表（fs）

```
cd <bridge>
env -u LD_LIBRARY_PATH PATH=$HOME/.cargo/bin:$PATH \
  cargo run --release --example provision_local -- /home/omm/iw_scalar
```

建 `default.t`（`id int`、`name string`、`score double`，5 行 id=1..5），`metadata_location` 写入 `/home/omm/iw_scalar/.metadata_location`。

### ④ 建标量 BTree 索引（旁路供数器）

catalog 暂不支持 btree，索引经 bridge C-ABI 直接构建：

```
cc -std=c11 -x c -I<bridge>/include <docs>/9-build_scalar_index.c \
   -L<bridge>/target/release -liceberg_rust_bridge \
   -Wl,-rpath,<bridge>/target/release -o build_scalar_index
env -u LD_LIBRARY_PATH ./build_scalar_index /home/omm/iw_scalar
```

输出 `INDEXED_METADATA=<带 idx_id 的 metadata>`，记为 `SC_META`。

### ⑤ 供数：向量表（fs）

```
cd <bridge>
env -u LD_LIBRARY_PATH PATH=$HOME/.cargo/bin:$PATH \
  ICEBERG_FILE_SCAN_WAREHOUSE=file:///home/omm/iw_vec \
  ICEBERG_FILE_SCAN_NAMESPACE=vscan_ns ICEBERG_FILE_SCAN_TABLE=scan_table \
  cargo test --release --test file_scan_vector file_scan_vector -- --nocapture
```

建 `vscan_ns.scan_table`（`c_integer int`、`c_vector list<float>` dim3，5 行；`[1,0,0]/[2,0,0]/[1,2,0]/[1,0,3]/[5,0,0]`）。记录打印的 `METADATA_LOCATION` 为 `base`。

### ⑥ 建向量 IVF 索引（旁路供数器）

```
cc -std=c11 -x c -I<bridge>/include <docs>/9-build_vector_index.c \
   -L<bridge>/target/release -liceberg_rust_bridge \
   -Wl,-rpath,<bridge>/target/release -o build_vector_index
env -u LD_LIBRARY_PATH ./build_vector_index '<base>' vscan_ns scan_table c_vector idx_vec
```

输出 `INDEXED_METADATA=<带 idx_vec 的 metadata>`，记为 `VEC_META`。

### ⑦ 接线并跑 2×2

```
gsql -d postgres -p 26000 -c "CREATE DATABASE twobytwo;"
gsql -d twobytwo -p 26000 \
  -v sc_meta="'$SC_META'" -v vec_meta="'$VEC_META'" \
  -f <docs>/9-twobytwo.sql
```

脚本在单一 `iceberg_catalog_server`（fs）下建 4 张外表（`demo.sc_full/sc_idx/vec_full/vec_idx`），将元数据指针改指供数表，对 `sc_idx`/`vec_idx` 加 mock 选项，逐条 EXPLAIN + 实扫。namespace/table 对 bridge 是未校验 label，真实定位靠 `metadata_location`（`StaticTable::from_metadata_file`），故外表命名自由。

## 6. 预期结果

| 路径 | EXPLAIN 关键行 | 行结果 |
|---|---|---|
| ① 标量×全表 | `bridge full scan` + `SDK Filter: present`（id>3, exact） | `4 bob` / `5 eve` |
| ② 标量×索引 | `bridge scalar index scan` / `Index Name: idx_id` | `4 bob` / `5 eve` |
| ③ 向量×全表 | `Limit → Vector Search(Algorithm TopK, Metric L2, TopK 3) → bridge full scan` | `c_integer` = 1,2,3 |
| ④ 向量×索引 | `bridge vector index scan` / `Index Name: idx_vec` / `TopK 3` | `1 [1,0,0]` / `2 [2,0,0]` / `3 [1,2,0]` |

③④ 同解（1,2,3）表明 VectorSearch-over-full-scan 与 IVF 索引路径等价；①② 同解（4,5）表明全表下推与 btree 索引一致。去掉外表 mock 选项后，②④ 回退 `bridge full scan` 返回全表。两索引路径的 ReScan 可经嵌套循环（如 `(VALUES(10),(20)) CROSS JOIN demo.sc_idx`）验证每轮结果一致。

## 7. 说明

- **存储无关**：bridge by_metadata 索引（向量 IVF、标量 btree）的 build/search 均经 `metadata_location` + `FileIO` 定位，fs/s3 皆可；本流程全程 `file://`，无需 MinIO/REST。
- **类型白名单**：FDW 外表列支持 `int2/int4/int8`、`text/varchar/bpchar`、`vector(n)`；`double precision` 不支持，标量表的 `score` 列在外表中省略。
- **snapshot**：by-metadata 标量搜索由 `metadata_location` 固定快照，不消费 `snapshot_id`。
- **mock 为临时脚手架**：`9-fdw-2x2-mock-shim.patch` 在规划期索引发现（标量谓词→`expression_json` 翻译、向量 `query_vector` 注入）就绪后移除。
